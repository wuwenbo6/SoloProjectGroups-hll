Bucket Policy IAM

Bucket policies control access to your S3 resources. Use Allow to grant or Deny to block specific actions. The policy evaluation order is: first matching Deny wins → then first matching Allow → default Allow (no policy).

S3 API Examples

# 1. Create Bucket
aws s3 mb s3://my-bucket --endpoint-url http://localhost:9000 --profile mini-rgw

# 2. List Buckets
aws s3 ls --endpoint-url http://localhost:9000 --profile mini-rgw

# 3. Upload Object (PutObject)
aws s3 cp ./file.txt s3://my-bucket/file.txt --endpoint-url http://localhost:9000 --profile mini-rgw

# 4. Download Object (GetObject)
aws s3 cp s3://my-bucket/file.txt ./downloaded.txt --endpoint-url http://localhost:9000 --profile mini-rgw

# 5. List Objects in Bucket
aws s3 ls s3://my-bucket/ --endpoint-url http://localhost:9000 --profile mini-rgw

# 6. Multipart Upload (large file auto-multipart by awscli)
aws s3 cp ./large-file.zip s3://my-bucket/large-file.zip --endpoint-url http://localhost:9000 --profile mini-rgw

Storage Layout: Tenant/Bucket Prefix

# Buckets are stored with tenant prefix on disk
data/buckets/tenant-a/my-bucket/object-key
data/buckets/tenant-b/my-bucket/object-key

# Same bucket name, different tenants → isolated storage
This matches Ceph RGW's multi-tenant bucket naming convention.

Multi-Tenant Explanation

This simulator supports multi-tenancy like Ceph RGW. Each tenant is identified by a unique Tenant ID and has its own Access Key / Secret Key pair. Different tenants have isolated namespaces — a bucket named "data" under tenant-a is completely separate from a bucket named "data" under tenant-b. The tenant mapping is determined by the Access Key used in the AWS Signature V4 authentication: each Access Key is bound to exactly one tenant.
